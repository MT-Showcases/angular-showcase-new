🎓 Perché i Signals?

Prima di Angular 17, lo stato reattivo si gestiva principalmente con RxJS Observable e BehaviorSubject. I Signals sono un'alternativa più semplice per lo stato locale: niente subscription, niente unsubscription, aggiornamenti automatici della UI.

✅ Usa Signals quando...

  • Lo stato è locale al componente
  • Hai valori derivati (computed) da aggiornare automaticamente
  • Vuoi evitare la complessità di RxJS per casi semplici
  • Lavori con Angular 17+ in zone-less mode

⚡ Usa RxJS quando...

  • Gestisci stream asincroni (HTTP, WebSocket)
  • Hai bisogno di operatori complessi (debounce, switchMap, combineLatest)
  • Lo stato va condiviso tra componenti via service
  • Lavori con NgRx Effects

💡 Pro tip: Puoi usare toSignal() e toObservable() per convertire tra i due mondi. Non sono in competizione — si completano!

🚫 Anti-Pattern: Loop Infinito

Non modificare un signal all'interno del suo stesso computed() o effect(). Questo causa un loop infinito e un errore a runtime.

// ❌ ANTI-PATTERN — loop infinito!
count = signal(0);
bad = computed(() => {
  this.count.set(this.count() + 1); // ← NON fare questo!
  return this.count();
});

// ✅ CORRETTO — computed è solo lettura
doubled = computed(() => this.count() * 2);

Writable Signal - Counter

Un writable signal è un signal che può essere modificato usando .set() o .update().

{{ count() }}
count = signal(0);
increment() {{ '{' }} this.count.update(v => v + 1); {{ '}' }}

Computed Signal - Valori Derivati

Un computed signal calcola automaticamente il suo valore basandosi su altri signal. Si aggiorna automaticamente quando i signal da cui dipende cambiano.

Count
{{ count() }}
Double
{{ doubleCount() }}
Triple
{{ tripleCount() }}
doubleCount = computed(() => this.count() * 2);
tripleCount = computed(() => this.count() * 3);

Esempio Pratico - Carrello

Carrello con signal per gli items e computed per il totale. Il totale si aggiorna automaticamente quando modifichi le quantità.

@for (item of items(); track item.name) {
{{ item.name }}
€{{ item.price }}
{{ item.quantity }}
}
Totale: €{{ total() }}
items = signal([...]);
total = computed(() => items().reduce(...));

Effect - Side Effects

Un effect viene eseguito automaticamente ogni volta che i signal da cui dipende cambiano. Perfetto per logging, analytics, o sincronizzazione.

Log Effect:
@for (log of effectLogs(); track log) {
{{ log }}
} @empty {
Nessun log ancora...
}
effect(() => {{ '{' }}
  console.log('Count:', this.count());
{{ '}' }});

⚖️ Signals vs RxJS — Quando usare cosa?

🔵 Signals

Stato locale, valori derivati, UI reattiva semplice

// Stato locale del componente
count = signal(0);
doubled = computed(() => this.count() * 2);

// Nessuna subscription!
increment() { this.count.update(v => v + 1); }

// Side effect automatico
effect(() => localStorage.setItem('count', this.count()));
  • ✅ Semplice da usare
  • ✅ Nessun memory leak (no unsubscribe)
  • ✅ Ottimo per stato locale
  • ✅ computed() è memoized automaticamente
  • ❌ Non adatto per stream asincroni
  • ❌ Nessun operatore (debounce, switchMap...)

🟣 RxJS Observable

Stream asincroni, HTTP, WebSocket, operatori complessi

// HTTP con operatori
search$ = this.searchTerm$.pipe(
  debounceTime(300),
  distinctUntilChanged(),
  switchMap(term => this.api.search(term))
);

// Conversione verso Signal
searchResults = toSignal(this.search$, { initialValue: [] });
  • ✅ Perfetto per HTTP e WebSocket
  • ✅ Operatori potenti (pipe, switchMap...)
  • ✅ Composizione di stream multipli
  • ✅ Integrazione con NgRx Effects
  • ❌ Richiede subscription/unsubscription
  • ❌ Curva di apprendimento più ripida

🔗 Il Ponte: toSignal() e toObservable()

Angular 16+ permette di convertire tra i due mondi in modo fluido:

import { toSignal, toObservable } from '@angular/core/rxjs-interop';

// Observable → Signal (gestione automatica unsubscribe)
posts = toSignal(this.postsService.getPosts(), { initialValue: [] });

// Signal → Observable (per operatori RxJS)
count$ = toObservable(this.count);
