🎓 NgRx vs BehaviorSubject vs Signals

Scegliere lo strumento giusto per la gestione dello stato è una delle decisioni più importanti in un'applicazione Angular. Ecco una guida rapida:

⚡ Signals

Stato locale al componente

  • Contatori, form, UI toggles
  • Valori derivati con computed()
  • Nessun boilerplate
count = signal(0);

🔵 BehaviorSubject

Stato condiviso leggero

  • AuthService, CartService
  • Pochi componenti, pochi dati
  • RxJS operators (pipe, map)
private state$ = new BehaviorSubject(init);

🟣 NgRx (questo)

Stato globale enterprise

  • App grandi, team grandi
  • Time-travel debugging
  • Actions, Reducers, Effects, Selectors
store.dispatch(increment());

🚫 Anti-Pattern: Stato locale in NgRx

❌ Sbagliato — metti il dropdown open/close nel NgRx store:

// In store: isDropdownOpen: boolean
store.dispatch(toggleDropdown());

✅ Corretto — usa una variabile locale:

// Nel componente
isDropdownOpen = signal(false);

Regola pratica: se lo stato non viene mai letto da un altro componente, non appartiene allo store.
